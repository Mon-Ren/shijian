# 工作流组合

> "政策和策略是党的生命，各级领导同志务必充分注意，万万不可粗心大意。"
> —— 毛泽东

方法论的威力不在于单独使用某一件思想武器，而在于**在正确的时机以正确的顺序组合使用**。本 skill 提供五条标准化工作流，定义 skill 的调用顺序、步骤间的数据传递格式和终止条件。

## Workflow 1：新项目启动

**适用场景：** 从零开始面对一个新任务/项目/领域，目标已知但路径未知，资源有限。

**触发信号：** 新项目、新领域、MVP、从零开始、不知道从哪里下手

### 工作流

```
investigation-first → contradiction-analysis → spark-prairie-fire → protracted-strategy
     调查研究        →     矛盾分析          →     星火燎原        →     持久战略
```

### 步骤详解

**Step 1：investigation-first（调查研究）**
- 目标：摸清现状，占有第一手材料
- 输出格式（传递给 Step 2）：
  ```
  调查结论：
  - 现状是：……
  - 关键约束是：……
  - 主要的未知项是：……
  - 已知的可依托基础是：……
  ```
- 终止条件：能够回答"现在是什么情况"这个问题

**Step 2：contradiction-analysis（矛盾分析）**
- 输入：Step 1 的调查结论
- 目标：从调查材料中识别主要矛盾，找到最值得突破的切入点
- 输出格式（传递给 Step 3）：
  ```
  主要矛盾：[A] vs [B]
  矛盾性质：[对抗性/非对抗性]
  最值得突破的切入点：……
  原因：解决了它，[X]、[Y] 问题随之缓解
  ```
- 终止条件：找到一个明确的主要矛盾和切入点

**Step 3：spark-prairie-fire（星火燎原）**
- 输入：Step 2 识别的切入点
- 目标：以切入点为根据地，制定从小到大的发展路线
- 输出格式（传递给 Step 4）：
  ```
  根据地：[具体切入点]
  发展路线：
    第1步：[具体行动] → 成功标志：……
    第2步：[具体行动] → 成功标志：……
    第3步：[具体行动] → 成功标志：……
  ```
- 终止条件：有一条清晰的、从当前能力出发的发展路线

**Step 4：protracted-strategy（持久战略）**
- 输入：Step 3 的发展路线
- 目标：将发展路线纳入阶段性战略框架，确定当前处于哪个阶段，设置阶段转换条件
- 输出格式：
  ```
  当前阶段：战略防御期
  本阶段核心任务：[Step 3 第1步]
  转入相持期的条件：当 [具体里程碑] 达成时
  ```
- 终止条件：有明确的阶段定位和转换条件

## Workflow 2：复杂问题攻坚

**适用场景：** 面对一个已知存在但难以解决的具体问题，问题边界清楚但原因不明或路径不清。

**触发信号：** 疑难 bug、架构瓶颈、反复失败的尝试、根因不明的问题

### 工作流

```
investigation-first → contradiction-analysis → concentrate-forces → practice-cognition → criticism-self-criticism
     调查研究        →     矛盾分析          →     集中兵力        →    实践认识论      →     批评与自我批评
```

### 步骤详解

**Step 1：investigation-first（调查研究）**
- 目标：弄清问题的真实面目（而非症状）
- 输出（传递给 Step 2）：
  ```
  调查结论：
  - 问题的现象是：……
  - 问题出现的条件是：……
  - 已经排除的原因是：……
  - 还未调查的方向是：……
  ```
- 终止条件：能够精确描述问题（而非"好像是 X 出了问题"）

**Step 2：contradiction-analysis（矛盾分析）**
- 输入：Step 1 的调查结论
- 目标：找到问题中的主要矛盾（根本冲突点）
- 输出（传递给 Step 3）：
  ```
  主要矛盾：[根本冲突点]
  我的假说：这个冲突导致了……，因为……
  验证方式：如果假说正确，应该能观察到……
  ```
- 终止条件：有一个可以验证的假说

**Step 3：concentrate-forces（集中兵力）**
- 输入：Step 2 的假说和验证方式
- 目标：围绕验证这个假说，集中全部力量，拒绝分心
- 输出（传递给 Step 4）：
  ```
  🎯 主攻目标：验证假说"[假说内容]"
  验证方法：[具体的验证步骤]
  完成信号：[假说被证实或证伪]
  暂缓的其他方向：……
  ```
- 终止条件：假说被验证（证实或证伪均可）

**Step 4：practice-cognition（实践认识论）**
- 输入：Step 3 的验证结果
- 目标：根据验证结果，更新认识，进入下一轮（如果假说被证伪则回到 Step 2）
- 循环终止条件：问题被彻底解决（可以明确描述原因和解决方案）

**Step 5：criticism-self-criticism（批评与自我批评）**
- 输入：整个攻坚过程
- 目标：审视攻坚过程中的方法论失误，防止下次犯同类错误
- 必须输出：结构化工作审视报告

## Workflow 3：方案迭代优化

**适用场景：** 已有一个基础方案，需要通过收集反馈、分析问题、迭代改进来提升质量。

**触发信号：** 迭代优化、用户反馈、代码审查后改进、方案效果不理想

### 工作流

```
mass-line → contradiction-analysis → practice-cognition → criticism-self-criticism → mass-line（循环）
 群众路线 →     矛盾分析          →    实践认识论      →     批评与自我批评      →  群众路线（下一轮）
```

### 步骤详解

**Step 1：mass-line（群众路线）**
- 目标：从多个信息源收集对现有方案的反馈
- 信息源清单（根据具体情况选择）：
  - 用户的明确反馈
  - 测试结果/运行结果
  - 代码审查意见
  - 性能数据/指标
  - 代码库中相关模式的启示
- 输出（传递给 Step 2）：
  ```
  收集到的多源反馈：
  - [信息源1]：……
  - [信息源2]：……
  各源之间的矛盾之处：……
  共同指向的问题：……
  ```
- 终止条件：有来自至少 2 个信息源的反馈

**Step 2：contradiction-analysis（矛盾分析）**
- 输入：Step 1 的多源反馈
- 目标：从反馈中识别出需要优先解决的主要矛盾
- 输出（传递给 Step 3）：
  ```
  本轮迭代的主要矛盾：[当前方案的核心问题]
  改进假说：如果……，则……
  验证方式：……
  ```

**Step 3：practice-cognition（实践认识论）**
- 输入：Step 2 的改进假说
- 目标：实施改进并验证效果
- 循环终止条件（开始前明确说明）：当 [具体指标] 达到 [具体数值/状态] 时，本轮迭代完成

**Step 4：criticism-self-criticism（批评与自我批评）**
- 目标：审视本轮迭代的过程质量
- 必须输出：工作审视报告

**Step 5：mass-line（下一轮）**
- 基于本轮的改进结果，重新收集反馈，开始新的迭代循环

**整体循环终止条件：** 当 mass-line 的多源反馈显示"没有新的重要问题"或用户确认满意时，退出循环。

## Workflow 4：紧急故障响应

**适用场景：** 系统出现紧急故障，需要快速定位根因并修复，不能按部就班走完整流程。时间压力大，容错空间小。

**触发信号：** 线上故障、服务挂了、P0/P1 级 bug、用户大面积报错、需要立即响应

**与 Workflow 2（复杂问题攻坚）的区别：** 攻坚是"已知问题但根因不明"，可以花时间深入分析；故障响应是"正在出血，先止血再查因"，速度优先。

### 工作流

```
investigation-first（精简版）→ contradiction-analysis（精简版）→ concentrate-forces → practice-cognition
       快速摸底                  →        锁定最可能原因        →    集中修复    →   验证效果
```

### 步骤详解

**Step 1：investigation-first · 快速摸底（限时 5 分钟内）**
- 目标：用最短时间获取关键事实，不追求全面，只抓要害
- 必查项（精简提纲）：
  - `- [ ]` 故障现象是什么？什么时候开始的？
  - `- [ ]` 最近有没有部署/配置变更？（git log、发布记录）
  - `- [ ]` 错误日志/监控的最新数据是什么？
  - `- [ ]` 影响范围多大？（哪些用户/功能受影响）
- 输出（传递给 Step 2）：
  ```
  快速摸底结论：
  - 故障现象 + 时间线：……
  - 最近变更记录：……
  - 最可疑的方向：……
  - 已排除的方向：……
  ```
- **禁止：** 在这一步花大量时间做全面调查。只看最关键的信号。

**Step 2：contradiction-analysis · 锁定最可能原因（限时 3 分钟内）**
- 输入：Step 1 的快速摸底结论
- 目标：不做完整矛盾分析，只基于已有事实锁定 1-2 个最可能的原因
- 输出（传递给 Step 3）：
  ```
  最可能原因（按概率排序）：
  1. [原因 A] —— 依据：……
  2. [原因 B] —— 依据：……
  验证优先级：先查 [A]，因为……
  ```
- **禁止：** 列出 5 个以上可能原因。只保留最有证据支撑的 1-2 个。

**Step 3：concentrate-forces · 集中修复**
- 输入：Step 2 锁定的最可能原因
- 目标：围绕最可能原因集中全部力量修复
- 执行规则：
  - 如果验证发现原因 A 正确 → 立即修复
  - 如果验证发现原因 A 不对 → 立即转到原因 B，不犹豫
  - 修复方案优先选择"最快止血"的方案，不要追求完美
- 输出（传递给 Step 4）：
  ```
  🎯 修复动作：[具体做了什么]
  修复时间：……
  止血方案 vs 根治方案：[如果用了止血方案，后续需要根治]
  ```

**Step 4：practice-cognition · 验证效果**
- 输入：Step 3 的修复动作
- 目标：确认故障是否已解决
- 必须做：
  - 检查监控/日志是否恢复正常
  - 确认受影响功能是否恢复
  - 观察至少 5 分钟确认没有复发
- 循环机制：如果修复无效 → 回到 Step 2，选择下一个最可能原因

**故障响应结束后（非紧急阶段）：**
- 用 shijian-criticism-self-criticism 做事后复盘（postmortem）
- 用 shijian-investigation-first 做根因分析（如果当时用了止血方案而非根治）
- 将复盘结论记录下来，防止同类故障复发

## Workflow 5：技术选型

**适用场景：** 面对多个可选方案（技术栈、架构设计、工具选型、实现路径等），需要系统化对比并做出决策。

**触发信号：** 选型、方案对比、技术决策、A vs B、多个方案选哪个

**核心原则：** 不靠直觉拍板，用事实和结构化对比做决策。

### 工作流

```
investigation-first → contradiction-analysis → overall-planning → practice-cognition → criticism-self-criticism
     需求调研        →     核心约束分析      →     多维对比权衡   →     原型验证       →      决策复盘
```

### 步骤详解

**Step 1：investigation-first · 需求调研**
- 目标：搞清楚"我们到底在选什么"以及"选的标准是什么"
- 调查提纲：
  - `- [ ]` 要解决的核心问题是什么？（不是"选什么"，是"为什么选"）
  - `- [ ]` 候选方案有哪些？各自的核心特性是什么？
  - `- [ ]` 团队/项目的约束条件？（技术栈、人员能力、预算、时间）
  - `- [ ]` 同类项目/社区的经验教训？（他人走过的弯路）
- 输出（传递给 Step 2）：
  ```
  选型需求：
  - 要解决的问题：……
  - 候选方案：[A]、[B]、[C]……
  - 硬性约束：……
  - 已知的他人经验/教训：……
  ```

**Step 2：contradiction-analysis · 核心约束分析**
- 输入：Step 1 的需求调研
- 目标：识别选型中最关键的 trade-off（主要矛盾），确定"最不能妥协"的维度
- 输出（传递给 Step 3）：
  ```
  选型中的主要矛盾：[维度A] vs [维度B]
  例如：性能 vs 开发效率、成熟度 vs 灵活性、功能丰富 vs 学习成本
  我方的优先级判断：在当前场景下，[维度X] 是最不能妥协的
  原因：……
  ```

**Step 3：overall-planning · 多维对比权衡**
- 输入：Step 2 的核心约束 + Step 1 的候选方案
- 目标：在多个评估维度之间做系统化对比，给出结构化决策
- 评估维度（根据具体情况增减）：
  - 功能匹配度：能否满足核心需求？
  - 技术成熟度：社区活跃度、文档质量、生产验证？
  - 学习成本：团队能否快速上手？
  - 可维护性：长期维护的代价？
  - 性能/资源：是否满足性能要求？
  - 生态兼容：与现有技术栈的兼容性？
- 输出格式：
  ```
  选型对比矩阵：
  | 维度 | 权重 | 方案A | 方案B | 方案C |
  |------|------|-------|-------|-------|
  | 功能匹配度 | 高 | ✅/⚠️/❌ | … | … |
  | 技术成熟度 | 中 | … | … | … |
  | … | … | … | … | … |

  推荐方案：[X]
  推荐理由（一句话）：……
  主要风险：……
  风险缓解措施：……
  ```
- **禁止：** 只说"A 好"但不给对比依据。必须有结构化的对比矩阵。

**Step 4：practice-cognition · 原型验证（可选但推荐）**
- 输入：Step 3 的推荐方案
- 目标：用最小成本验证推荐方案是否真的可行
- 做法：
  - 搭建一个最小原型 / PoC（Proof of Concept）
  - 跑通核心场景
  - 观察实际体验 vs 理论预期的差距
- 输出：
  ```
  原型验证结论：
  - 验证了什么：……
  - 实际结果 vs 预期：……
  - 发现的问题：……
  - 结论：[推荐方案确认 / 推荐方案需要调整 / 需要重新选型]
  ```

**Step 5：criticism-self-criticism · 决策复盘**
- 目标：审视选型过程是否有遗漏或偏见
- 自查问题：
  - 有没有被某个方案的宣传材料影响而忽略了实际验证？
  - 有没有因为熟悉某个方案而产生偏好偏差？
  - 评估维度是否完整？有没有遗漏重要的考量因素？
  - 对比矩阵中的判断是否有事实依据还是凭感觉？
- 必须输出：工作审视报告（重点放在"决策过程的质量"而非"工作完成情况"）

## 工作流选择指南

| 你的情况 | 推荐工作流 |
|---------|---------|
| 面对全新的项目/任务，不知道从哪里下手 | Workflow 1：新项目启动 |
| 有一个具体的疑难问题，需要找到根因并解决 | Workflow 2：复杂问题攻坚 |
| 已有方案但效果不理想，需要迭代优化 | Workflow 3：方案迭代优化 |
| 线上紧急故障，需要快速止血 | Workflow 4：紧急故障响应 |
| 多个方案需要对比选型 | Workflow 5：技术选型 |
| 以上都不完全符合 | 回到 shijian-arming-thought 的调度表，单独调用最匹配的 skill |

## 数据传递规则

工作流中每个步骤的**输出必须显式传递给下一步骤**。具体做法：

1. 每个步骤结束时，写出格式化的"传递给下一步的信息"
2. 下一步开始时，确认收到并引用上一步的输出
3. 不得跳步骤执行（如不得跳过 contradiction-analysis 直接进入 concentrate-forces）

## 工作流中断处理

如果在工作流执行过程中遇到阻塞：
- **信息不足** → 插入一次 shijian-investigation-first，补充信息后继续
- **假说被证伪** → 返回 shijian-contradiction-analysis，重新分析
- **发现新的更重要的矛盾** → 用 shijian-overall-planning 评估是否需要调整工作流
