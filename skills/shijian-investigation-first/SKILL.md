# 调查研究

> "没有调查，没有发言权。"
> —— 毛泽东《反对本本主义》（1930年）

## 核心原则

**一切正确的决策都来源于对实际情况的调查研究。没有调查就没有发言权，没有正确的调查同样没有发言权。**

在本项目的方法体系中，`调查研究` 是落实 `实事求是` 的主要入口。调查不是为既定结论搜集材料，而是让事实来规定判断，让现实来修正理论。

> 详细原著依据见 original-texts.md

## 不适用场景

**以下情况不需要调用此 skill：**
- 当前任务是已充分调查后的执行阶段 —— 不要重复调查
- 用户已经提供了完整的上下文和决策依据
- 问题是技术实现细节，答案可以直接验证（运行代码看结果）
- 紧急的修复任务且问题现象已经明确 —— 直接定位和修复，调查会浪费时间
- 已有足够信息但需要多源交叉验证 → 用 shijian-mass-line 而非重新调查

## 何时使用

你应该在以下情况调用此 skill：

- 要做一个决策，但对实际情况了解不够
- 进入一个陌生的领域或面对一个新问题
- 发现自己在凭"直觉"或"经验"做判断，缺乏事实依据
- 别人的方案/建议听起来合理，但你不了解其适用条件
- 需要全面了解一个系统/项目/问题的现状
- 感觉自己在"本本主义"——套用理论而不了解实际

## 方法流程

### 第一步：明确调查目的

> "调查就像'十月怀胎'，解决问题就像'一朝分娩'。"——《反对本本主义》

调查不是漫无目的地搜集信息。在开始之前明确：
- 我要解决什么问题？
- 我需要了解哪些方面的情况？
- 调查的结果将如何指导我的行动？

### 第二步：选择场景，执行工具调用模板

> "你对于某个问题没有调查，就停止你对于某个问题的发言权。"——《反对本本主义》

**核心机制：本 skill 的调查不依赖 AI 自行判断"该看什么"，而是按预置的工具调用模板自动执行。** 每个模板包含具体的 shell 命令或文件读取操作，AI 的职责是：选择正确的场景 → 执行模板中的每个命令 → 如实记录输出。

**警惕：**
> 不能搞"走马看花"式的调查——模板中的每一项都必须执行，不得跳过或用"大致了解"替代。

#### 如何选择场景

| 场景 | 代码 | 典型信号 |
|------|------|---------|
| 代码/功能类 | `a` | bug、逻辑不清、功能异常 |
| 配置/环境类 | `b` | 部署失败、环境差异、配置错误 |
| 性能/资源类 | `c` | 响应慢、内存/CPU 异常、超时 |
| 架构/设计类 | `d` | 理解系统、评估方案、找技术债 |
| 需求/业务类 | `e` | 理解需求、评估优先级、梳理逻辑 |

如果任务跨多个场景，**每个场景的模板都必须执行**。

#### 场景 A：代码/功能类 —— 工具调用模板

> 适用：调试 bug、理解代码逻辑、排查功能异常

**每一步都必须执行工具调用，将原始输出填入"调查结果"列。禁止只写结论不附输出。**

| # | 调查项 | 工具调用（直接执行） | 调查结果（填写原始输出摘要） |
|---|--------|-------------------|--------------------------|
| 1 | 定位项目结构 | `find . -maxdepth 2 -not -path './.git/*' -not -path './node_modules/*'` | `[实际输出]` |
| 2 | 查看最近变更 | `git log --oneline -15` | `[实际输出]` |
| 3 | 检查最近 diff | `git diff HEAD~3 --stat 2>/dev/null \|\| git diff HEAD --stat` | `[实际输出]` |
| 4 | 读取入口文件 | `read: [根据步骤 1 确定的入口文件路径]` | `[文件内容摘要，含关键行号]` |
| 5 | 查找错误信号 | `grep -rn "error\|Error\|ERROR\|FIXME\|TODO\|HACK\|XXX" --include="*.{js,ts,py,go,rs,java,rb,sh}" . 2>/dev/null \| head -30` | `[实际输出]` |
| 6 | 搜索问题相关代码 | `grep -rn "[用户问题中的关键词]" --include="*.{扩展名}" . 2>/dev/null \| head -20` | `[实际输出]` |
| 7 | 运行测试（如有） | `npm test 2>&1 \| tail -30` 或 `pytest -x 2>&1 \| tail -30` 或 `go test ./... 2>&1 \| tail -30` | `[实际输出]` |

**执行顺序：** 1→2→3→4→5→6→7。步骤 4 的文件路径由步骤 1 的输出决定，步骤 6 的关键词由调查问题决定。

### 第三步：基于工具输出，系统整理

> "要自己做记录"——《反对本本主义》

**这一步的操作对象是上一步的工具调用输出，不是 AI 的"印象"。**

整理要求：
- 从工具输出中提取事实，不是从记忆中回忆
- 区分事实（工具输出的原始数据）与观点（AI 的解读）
- 整理归类，找出规律
- 标注存疑的地方

### 第四步：实事求是，得出结论

> "'实事'就是客观存在着的一切事物，'是'就是客观事物的内部联系，即规律性，'求'就是我们去研究。"——《改造我们的学习》

从调查得到的事实出发，分析其内在联系，得出结论：
- 事实先于判断，材料先于观点
- 结论必须有事实支撑
- 不能先有结论再找证据
- 如果事实与预期不符，修改的应该是结论而非事实
- 承认"不知道"也是一种诚实的调查结论

### 第五步：调查完成后再做决策

调查结束后，基于调查结果做出决策或提出方案。此时再与理论原则结合——**但理论必须服从事实，不是事实服从理论**。如果后续进入验证和迭代，继续调用 `shijian-practice-cognition`，让 `实事求是` 贯穿从调查到实践检验的全过程。

## 调查的七项注意

根据毛泽东在《反对本本主义》中提出的调查方法，整理为七项注意：

1. **要有目的** —— 带着问题去调查
2. **要有提纲** —— 调查有计划、有条理（场景模板即提纲）
3. **要亲自做** —— 执行工具调用，获取原始输出，不靠猜
4. **要深入** —— 模板中的每一步都执行，不跳过
5. **要做记录** —— 工具输出就是记录，结论必须引用输出
6. **要多方面** —— 跨场景时，多个模板都执行
7. **要实事求是** —— 从事实出发（工具输出），不预设结论

## 常见错误

| 错误 | 毛泽东的批评 | 正确做法 |
|------|------------|---------|
| 不调查就给出答案 | "没有调查，没有发言权" | 先执行模板，后发言 |
| 只看文档不看实际 | "不注重研究现状" | 执行工具调用获取第一手数据 |
| 调查走马观花 | 不能"站在高山上远观" | 模板每步都执行，不跳过 |
| 先有结论再找证据 | "不是从客观存在着的实际事物出发" | 先执行，再分析输出 |
| 只调查现状不看历史 | "不注重研究历史" | git log 必须执行 |
| 只看一方面 | 需要多角度多方面的调查 | 跨场景时多模板执行 |

## 操作规程

当本 skill 被触发时，**按以下顺序执行**：

1. **声明调查目的**：一句话写出"我要弄清楚的是：[具体问题]"

2. **选择场景并执行工具调用模板**：
   - 判断属于哪个场景（a/b/c/d/e），选择对应模板
   - **逐行执行模板中的工具调用命令**，将命令的实际输出填入"调查结果"列
   - 跨场景时，每个场景的模板都要执行
   - 如果模板中的命令不适用（比如没有 git 仓库），标注 `[不适用]` 并说明原因，不得跳过

3. **基于工具输出整理调查结论**，用固定格式收尾：
   ```
   调查结论：
   - 现状是：……
   - 关键约束是：……
   - 我之前不知道但现在知道的是：……
   - 基于以上，我的判断是：……

   事实锚点（来自工具调用输出）：
   - 项目结构：[来自场景 X 步骤 1 的输出]
   - 最近变更：[来自场景 X 步骤 2 的 git log 输出]
   - 关键文件：[来自场景 X 步骤 4 的 read 输出]
   - 错误/问题信号：[来自场景 X 步骤 5 的 grep 输出]
   ```

   **判定规则：**
   - ✅ 通过：事实锚点中引用了具体的工具输出内容
   - ❌ 空转：事实锚点为空，或只写了"已调查"但无具体输出 → 必须重新执行模板
   - ❌ 编造：锚点中的输出与实际执行结果不符 → 判定为未真实调查

**调查结论必须先于任何行动方案出现，不得边调查边给方案。**
